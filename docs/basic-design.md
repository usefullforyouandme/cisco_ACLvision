- 更新日: 2026-07-14

---

## 1. 概要

### 1.1 目的
Cisco 装置の `show running-config` 出力(全体または ACL 部分の抜粋)をインプットとして、
装置設定・ACL をブラウザ上で構造化・可視化し、人間によるレビュー・監査を効率化する Web ツールを提供する。

### 1.2 背景・課題
- show run のテキストを目視で追う ACL レビューは、object-group の展開・適用箇所の特定・
  ルール間の関係(シャドウ/冗長)の把握に多大な工数がかかり、見落としリスクも高い。
- 既存の商用ツール(FW ポリシー監査製品等)は高価・導入障壁が高く、
  「手元のコンフィグを貼り付けてすぐ確認したい」という軽量なニーズに合わない。

### 1.3 名称
**ACLvision**(仮)

## 2. 対象範囲

### 2.1 対象 OS(パーサ対応範囲)
| OS | 優先度 | 備考 |
|---|---|---|
| IOS / IOS-XE | 高(Phase 1) | numbered / named、standard / extended ACL |
| ASA | 中(Phase 2) | access-list 構文、object / object-group 展開 |
| NX-OS / FTD 等 | 対象外(将来拡張) | パーサ層の分離により追加可能な構造とする |

### 2.2 対象とする設定要素
**ACL レビュー・監査(主目的)**
- IOS: `access-list <num>`、`ip access-list standard|extended <name>`、`object-group`(network / service)
- IOS: 適用箇所 — `ip access-group`(interface)、`access-class`(line vty)、`snmp-server community <name> <acl>` 等の参照
- ASA: `access-list`(extended / remark)、`object network|service`、`object-group`、`access-group`(interface binding)

**全体設定の俯瞰(副目的)**
- ホスト名・OS バージョン・機種情報
- interface(IP アドレス、VLAN、description、shutdown 状態、適用 ACL)
- 管理系サービスの状態(SSH/Telnet、HTTP server、SNMP 等)— 俯瞰表示レベル
- 未知の設定行は「未解析行」として原文保持・表示(パース失敗で全体が落ちない)

### 2.3 対象外(v1 時点)
- IPv6 ACL(v1 は IPv4 のみ。IR の `AddressSpec` は IPv6 を追加可能な構造とし、将来拡張で対応)
- 通信可否シミュレーション(パケットトレース相当)— 将来拡張候補
- NAT・VPN・ルーティングプロトコル詳細の解析
- 装置間比較・差分表示(複数装置間の ACL 差分、変更前後の show run 差分)— 将来拡張候補
- 装置への接続・設定変更(本ツールは閲覧専用、読み取ったデータの書き戻しなし)

### 2.4 想定ユーザー
- セキュリティ診断・設定監査の実施者(コンサルタント)
- ネットワーク運用者(自組織の設定確認)

## 3. システム方式

### 3.1 アーキテクチャ方針
**完全クライアントサイド処理の静的 SPA**。設定データはブラウザ外に一切送信しない。

```
┌─ GitHub リポジトリ ─────────────────────────┐
│  ソースコード → GitHub Actions(build)→ GitHub Pages(静的配信) │
└──────────────────────────────────────┘
                     │ HTML/JS/CSS の配信のみ
                     ▼
┌─ ユーザーのブラウザ(すべての処理がここで完結)──────┐
│  入力層(貼り付け / ファイル読込)                    │
│    → パーサ層(OS 判定 → OS別パーサ → 共通データモデル) │
│    → 分析層(object-group 展開、参照解決、ルール分析)   │
│    → 表示層(各ビュー)                            │
└──────────────────────────────────────┘
```

- show run が機密情報(パスワードハッシュ、SNMP コミュニティ、内部 IP 体系等)を含む前提のため、
  「データを外に出さない」ことを最重要のアーキテクチャ制約とする(§8.1)。

### 3.2 技術スタック
| 項目 | 選定 | 理由 |
|---|---|---|
| フレームワーク | React 18 + TypeScript | パーサ・UI 双方の複雑化に耐える。型でデータモデルを厳密に定義 |
| ビルド | Vite | GitHub Actions → Pages のデプロイが容易 |
| 状態管理 | React 標準(Context / useReducer)から開始 | v1 の規模では外部ライブラリ不要。肥大化したら Zustand 等を検討 |
| スタイル | Tailwind CSS | ビルド時に静的 CSS 化されるためランタイム外部依存なしの制約(§8.1)と両立。開発速度を優先 |
| 図表描画 | SVG 直接描画(必要になったら軽量ライブラリ検討) | 外部依存を最小化。ランタイムの CDN 読込は禁止(§8.1) |
| テスト | Vitest(パーサのユニットテスト重点) | パーサは実コンフィグ断片をフィクスチャにしてテスト駆動で開発 |
| ホスティング | GitHub Pages(GitHub Actions でデプロイ) | 無料・運用レス。※public リポジトリ前提(§8.4) |

### 3.3 パーサ層の設計方針
1. **OS 自動判定**: 入力テキストの特徴から IOS / ASA を判定(例: `ASA Version` 行、
   `access-list ~ extended permit` 構文は ASA、`ip access-list extended` は IOS)。
   誤判定に備え手動切替も可能とする。
2. **OS 別パーサ → 共通中間モデル(IR)**: 各 OS の方言を吸収し、表示・分析層は IR のみに依存。
   将来の NX-OS 等の追加はパーサ追加のみで対応可能。
3. **寛容なパース**: 解析できない行はエラーにせず「未解析行」として保持し UI で明示。
   部分的な抜粋(ACL 部分のみの貼り付け)でも動作する。
4. **行番号の保持**: すべての解析結果は原文の行番号を保持し、UI から原文該当行へジャンプ可能。

### 3.4 共通データモデル(IR)概略
```typescript
Device {
  hostname, os: 'ios' | 'asa', version?, rawLines: string[]
  interfaces: Interface[]      // name, ip, vlan, description, shutdown, aclBindings
  acls: Acl[]                  // name/number, type(std/ext), rules: AclRule[]
  objectGroups: ObjectGroup[]  // network / service / protocol
  services: ServiceSummary     // ssh, telnet, http, snmp などの俯瞰情報
  unparsedLines: {lineNo, text}[]
}
AclRule {
  seq?, action: 'permit' | 'deny', protocol,
  src: AddressSpec, srcPort?: PortSpec,
  dst: AddressSpec, dstPort?: PortSpec,
  options: { log?, established?, ... },
  remark?: string[],            // 直前の remark 行を紐付け
  sourceLineNo
}
AddressSpec = any | host | subnet(addr/wildcard|mask) | object-group参照
```
- object-group は参照のまま保持し、表示時に展開(展開前後を切替可能に)。
- ACL と適用箇所(interface / line / snmp 等)の参照関係を解決し、
  「未使用 ACL」「未定義参照」を導出する。

## 4. 機能設計

### 4.1 入力機能
| 機能 | 内容 |
|---|---|
| テキスト貼り付け | show run 全体・ACL 部分抜粋のどちらも受付 |
| ファイル読込 | ドラッグ&ドロップ / ファイル選択。複数ファイル(複数装置)同時読込可 |
| 文字コード | UTF-8 / Shift_JIS を想定(description 等に日本語が含まれるケース) |
| データ消去 | ワンクリックで読込データを全消去する明示的なボタンを常設 |

### 4.2 可視化ビュー

#### V1: 装置サマリ(俯瞰)
- ホスト名・OS・バージョン、interface 一覧(IP / VLAN / 状態 / description / 適用 ACL)、
  ACL 一覧(名前 / 種別 / ルール数 / 適用箇所数)、管理サービス状態
- 統計: permit/deny 比率、`permit ip any any` 等の要注意ルール数のハイライト

#### V2: ACL テーブル(レビューの中核)
正規化されたルール表。1 行 = 1 ルールで以下を列表示:
`# | action | protocol | src | src-port | dst | dst-port | options | remark`
- **action の色分け**(permit=緑系 / deny=赤系)、`any` の強調表示
- **object-group のインライン展開切替**(参照名表示 ⇔ 実アドレス/ポート展開表示)
- remark をルールに紐付けて表示
- 行クリックで原文(該当行ハイライト)へジャンプ
- フィルタ: action / protocol / IP アドレス含有(例: 10.1.2.3 を含むルールに絞込 —
  サブネット包含判定)/ フリーテキスト

#### V3: 適用マップ
- 装置を中心に interface と ACL の適用関係(in/out)を SVG 図で表示
- ACL 側からは「どこに適用されているか」、interface 側からは「何が適用されているか」を相互参照
- 未使用 ACL・未定義参照(存在しない ACL を適用している)を警告表示

#### V4: レビュー支援(自動指摘)
機械的に検出可能な項目をチェックリスト形式で提示(判断は人間が行う前提):
| 検出項目 | 内容 |
|---|---|
| 過大許可 | `permit ip any any` 等、src/dst/port がすべて any のルール |
| シャドウルール | 先行ルールに完全に包含され、絶対に評価されないルール |
| 冗長ルール | 同一・包含関係にある同 action のルール |

シャドウ / 冗長検出は 2 段階で実装する(Phase 4):
1. object-group 展開前の名前ベース・アドレス直指定の包含判定(軽量、常時実行)
2. object-group 展開後の包含判定(計算量が増えるため、展開後の実効ルール数に上限を設け、
   超過時は「対象外(手動確認要)」と明示する)
| 未使用 ACL | 定義されているがどこにも適用されていない ACL |
| 未定義参照 | 適用されているが定義が存在しない ACL / object-group |
| 暗黙 deny の明示 | ACL 末尾の implicit deny を UI 上に明示表示。deny + log の有無も表示 |

### 4.3 複数装置の取り扱い
- 複数装置を同時に読み込み、装置タブ / 一覧で切替
- 装置間の ACL 比較(差分表示)は v1 対象外・将来拡張候補(§2.3)

### 4.4 エクスポート(監査報告書作成の支援)
- **Markdown エクスポートを最優先**(Phase 4)。正規化 ACL テーブル・レビュー支援(V4)の
  指摘一覧を Markdown 表として出力し、Markdown ベースの報告書にそのまま組み込める形式とする
- CSV エクスポート・印刷用 CSS(PDF 化)は Markdown 対応後の追加候補(実装コストが低いため
  余力があれば同 Phase 内で対応)

## 6. 処理フロー

1. 入力受付(貼り付け / ファイル)
2. OS 判定(自動、手動上書き可)
3. 行単位トークナイズ → OS 別パーサで IR 構築(未解析行は保持)
4. 参照解決(ACL⇔適用箇所、object-group)・分析(V4 の検出項目)
5. 各ビューへ描画。データはメモリ上のみ(永続化は §10 で要検討)

## 8. 非機能要件

### 8.1 セキュリティ(最重要)
- **設定データの外部送信を一切行わない**(fetch/XHR/WebSocket でのデータ送信なし。
  アナリティクス・外部フォント・ランタイム CDN 読込も不使用。全アセットを自己完結バンドル)
- CSP(Content-Security-Policy)メタタグで外部接続を技術的にも制限し、方針を検証可能にする
- **データ永続化は行わない**(localStorage / IndexedDB 等への保存なし。読込データはメモリ上のみで
  リロードにより消失する)。機密データのブラウザ残留リスクを排除することを利便性より優先する
- ページ上に「データはブラウザ内でのみ処理され、送信されません」の明示とデータ消去ボタン

### 8.2 性能
- 数万行規模の show run(大規模 ACL 含む)を数秒以内にパース・描画
- 大規模 ACL テーブルは仮想スクロールで描画負荷を抑制

### 8.3 対応ブラウザ・UI
- モダンブラウザ(Chrome / Edge / Firefox / Safari 最新)
- UI 言語: 日本語
- レスポンシブは PC 優先(レビュー業務は大画面前提)、印刷用スタイルを用意

### 8.4 リポジトリ・配布形態
- GitHub Pages(無料プラン)は public リポジトリが前提。**ツール自体は公開されるが、
  設定データはリポジトリ・サーバに一切含まれない/送られない**構成のため問題なし
- 補完案: `vite-plugin-singlefile` 等による**単一 HTML ファイル版**(オフライン環境・
  顧客先環境へのファイル配布用)は Phase 4 以降に後回しとする。ビルド構成上いつでも
  追加可能なため、オフライン利用の需要が顕在化した時点で対応する

## 9. 代替案の検討(ホスティング)

| 案 | 評価 |
|---|---|
| **GitHub Pages(採用)** | 無料・運用レス・Actions 連携。完全クライアント処理なら機密面も問題なし |
| 単一 HTML ファイル配布 | Pages と併用可能。オフライン・持ち込み環境で有効。Phase 4 以降に併用検討 |
| Cloudflare Pages / Netlify | private リポジトリでも無料枠あり。Pages で不足が出たら移行先候補 |
| ローカル CLI + レポート生成 | ブラウザ UI の対話性が失われるため不採用(将来、CI 組込み用に検討余地) |
